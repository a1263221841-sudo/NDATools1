# AI项目协作需求模板

这份文档用于帮助你在一个新项目开始时，清晰地向 AI 描述需求。

适用场景：
- 从 0 开始创建一个新项目
- 需要 AI 帮你设计项目结构、页面、模块和开发计划
- 需要 AI 分阶段实现代码、UI、配置、日志、主题、打包等功能
- 希望以后做其他项目时也能复用同一种表达方式

---

## 一、最推荐的提需求顺序

当你希望 AI 完整协助你搭建项目时，建议按下面顺序描述：

1. 项目目标
2. 开发技术
3. 功能模块
4. 界面设计
5. 业务规则
6. 代码要求
7. 交付方式
8. 验收标准

这样做的好处：
- AI 更容易理解你的真实目标
- 能减少返工
- 更适合分阶段推进
- 更方便后续继续迭代

## 三、最实用的协作方式

最推荐的开发流程不是一上来就说“帮我做完”，而是分成下面几个阶段。

### 阶段1：先做项目蓝图

你可以这样说：

```text
我要从 0 开始做一个项目，请先不要写代码，先帮我完成以下内容：
1. 项目结构设计
2. 页面划分
3. 类划分
4. UI布局建议
5. 公共模块设计
6. 分阶段开发计划
```

这一阶段的目的：
- 先把项目边界定清楚
- 避免后面越做越乱
- 让后续编码更稳定

### 阶段2：先搭项目骨架

你可以这样说：

```text
现在开始第 1 步，请你帮我搭建项目骨架：
- 创建主窗口
- 创建基础工程文件
- 创建公共工具类
- 创建空白页面
- 保证工程先能编译通过
```

这一阶段的目的：
- 先建立一个可扩展的工程框架
- 后面加功能不会太乱

### 阶段3：逐个模块实现

你可以这样说：

```text
现在请实现第一个模块：TCP服务器页面。
要求：
- 按当前项目风格设计 UI
- 补齐 .h / .cpp / .ui
- 完成信号槽连接
- 实现日志区、参数区、按钮逻辑
- 修改后检查诊断错误
```

这一阶段的目的：
- 一次只解决一个模块
- 更容易定位问题
- 每步都能验证效果

### 阶段4：做全局能力

你可以这样说：

```text
现在请统一处理整个项目的全局功能：
- 深色/浅色主题切换
- 系统托盘
- 开机自启
- 配置持久化
- 日志保存
- 界面最小尺寸和按钮可见性检查
```

这一阶段的目的：
- 给项目补完整性
- 让项目更接近可交付状态

### 阶段5：编译、联调、修问题

你可以这样说：

```text
现在请继续帮我收尾：
- 检查编译错误
- 修复运行问题
- 检查布局遮挡
- 检查按钮是否可点击
- 检查日志和配置是否正常
```

这一阶段的目的：
- 从“能写代码”进入“能运行”
- 从“能运行”进入“能使用”

## 四、如何描述一个模块

如果你不是想描述整个项目，而是只想让我实现一个页面或一个模块，可以按这个方式说。

```text
请帮我实现一个模块：HTTP测试页面。

模块目标：
- 发送 HTTP/HTTPS 请求
- 显示响应头和响应体

界面要求：
- 左侧显示响应结果
- 右侧显示请求参数
- 底部或右侧显示日志
- 风格仿照项目现有页面

功能要求：
- 支持 GET / POST / PUT / DELETE
- 支持自定义请求头
- 支持请求体输入
- 支持保存日志
- 支持自动记住上次输入的 URL 和 Header

代码要求：
- 使用 Qt Widgets
- 保持 .h/.cpp 分离
- 修改后检查诊断错误
```

## 五、如何描述界面要求

很多人提需求时最容易漏掉的，就是“界面怎么排”。

建议按下面方式说：

```text
界面要求：
- 主窗口使用顶部 Tab 标签页
- 左侧区域显示消息列表
- 右侧区域显示参数和按钮
- 页面宽度不足时，右侧参数区允许滚动
- 主题支持深色和浅色切换
- 按钮高度不要过小，避免点击困难
- 默认窗口大小要比设计图略大，避免控件被挡住
```

这样 AI 就更容易帮你避免布局问题。

---

## 六、如何描述业务规则

如果你只说“加个按钮”，通常是不够的。

更好的方式是把按钮点下去之后的行为说清楚。

例如：

```text
业务规则：
- 点击“连接”按钮后，先校验输入参数
- 参数为空时弹出提示
- 连接成功后写日志
- 连接失败时写错误日志并提示用户
- 连接期间禁用重复点击按钮
- 断开连接后恢复按钮状态
```

这样的描述更容易得到可运行代码，而不是只有界面。

---

## 七、如何描述代码风格

如果你想让项目长期维护更舒服，建议把这些要求提前说清楚：

```text
代码要求：
- 变量命名统一
- 函数命名尽量语义明确
- 注释使用中文
- 公共逻辑抽到工具类
- UI逻辑和业务逻辑尽量分开
- 修改后顺手检查诊断错误
- 遇到编译错误继续修到通过
```

## 八、适合长期协作的提问方式

下面是非常适合长期做项目的方式。

### 方式1：先设计再实现

```text
先不要写代码，请先帮我做项目设计文档。
我要确认项目结构、页面划分和开发步骤。
```

### 方式2：每次只推进一个阶段

```text
现在只做第一阶段：创建主窗口和空白页面。
不要提前做后面的功能。
```

### 方式3：每次只改一个模块

```text
现在只实现 MQTT 页面，其它页面先不要动。
做完后告诉我改了哪些文件。
```

### 方式4：发现报错后继续贴给 AI

```text
这是新的编译错误，请继续修改，直到能编译通过。
```

这种方式非常适合真实开发。

## 九、一个完整示例

下面给出一个完整示例，适合参考。

```text
我想从 0 开始做一个 Qt Widgets + C++ 的桌面网络调试工具，请你完整协助我开发。

一、项目目标
- 项目名称：NDA Tools 网络调试助手
- 目标用户：开发人员、测试人员、嵌入式调试人员
- 核心用途：做 TCP、UDP、HTTP、WebSocket、MQTT、Modbus TCP 等协议测试，同时集成数据转换和数据校验功能

二、开发技术
- 开发语言：C++
- 框架：Qt Widgets
- 工程类型：qmake .pro
- Qt版本：Qt 6.x
- 编译器：MinGW 64
- 使用 Qt Designer 的 .ui 文件

三、功能模块
1. TCP服务器
- 启动监听
- 接收客户端消息
- 发送测试消息
- 保存日志

2. TCP客户端
- 连接服务器
- 发送消息
- 接收消息
- 断开连接

3. UDP服务器
- 监听端口
- 接收消息
- 回复消息

4. UDP客户端
- 发送消息
- 显示返回结果

5. 数据处理工具
- 字符串转 Hex
- Hex 转字符串
- Base64 编码解码
- URL 编码解码

6. 数据校验工具
- CRC16
- CRC32
- MD5
- SHA1
- SHA256
- 支持复制结果
- 支持大小写切换
- 支持异步计算

7. 协议测试工具
- HTTP/HTTPS
- WebSocket
- POP3/IMAP
- MQTT
- Modbus TCP

8. 全局功能
- 系统托盘
- 开机自启
- 深色/浅色主题切换
- 配置持久化
- 日志保存到文件

四、界面设计
- 主窗口使用顶部 Tab 标签页切换
- 每个功能页尽量仿照传统调试工具风格
- 左侧显示消息列表或结果区
- 右侧显示参数和操作按钮
- 需要深色/浅色主题切换按钮
- 主题切换要作用于整个应用
- 协议页右侧内容较多时要支持滚动，避免按钮被挡住

五、业务细节
- 所有网络页面都需要日志区
- 所有地址、端口、主题、URL 等参数需要自动保存并恢复
- 大数据量计算时要异步处理，避免界面卡死
- 输入为空时要给出提示
- 错误时弹框或日志提示
- 普通提示尽量不打断用户操作

六、代码要求
- 保持 .h/.cpp/.ui/.pro 结构清晰
- 命名尽量统一
- 注释写中文
- 兼容 Qt5/Qt6 常见差异
- 修改后帮我检查诊断错误
- 如果出现编译错误，继续帮我修到能跑通

七、交付方式
- 第一步：先帮我创建项目骨架和主窗口
- 第二步：逐个模块实现
- 第三步：统一主题、日志、配置持久化
- 第四步：检查界面布局和编译问题
- 每一步都告诉我改了哪些文件，以及下一步建议
```

## 十、最简提问公式

如果你不想每次都写很多，可以记住下面这句话：

```text
我要做一个【什么项目】，
使用【什么技术】，
包含【哪些模块】，
界面要求是【什么布局/风格】，
业务规则是【按钮做什么、日志怎么记、异常怎么提示】，
代码要求是【命名/注释/兼容性】，
请按【先设计 / 先骨架 / 先某模块】的方式帮我一步一步完成。
```

---

## 十一、建议你以后这样用

建议你的协作节奏始终保持为：

1. 先明确目标
2. 先出设计方案
3. 再搭项目骨架
4. 再逐页实现
5. 最后统一修编译和布局

如果项目大，就永远不要一次性说“全部做完”，而是分阶段推进。

这样最好维护，也最容易成功。
